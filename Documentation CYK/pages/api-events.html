Game Events



    This section is all about game events. Game events are functions in your scripts that the CYF calls at certain points during an encounter.

    NOTICE: Entity and Wave Events have been moved into their respective pages!


    This page will primarily focus on the Events relevant to the main gameplay loop.

    Encounter script events


    Update() This function runs for every frame for all of the encounter, even during waves. For advanced users: the Update function is required to exist at the beginning of the encounter. If you define it at a later point in time, it will not get executed.



    EncounterStarting() Happens once when everything's done initializing but before any encounter actions start. You should do things like stopping the music here, or using State() if you want to start the fight off with some dialogue.



    EnemyDialogueStarting() Happens when you go to the monster dialogue state. You're still free to modify monster dialogue here.



    EnemyDialogueEnding() Happens when you go from the monster dialogue state to the defending state.



    DefenseEnding() Happens when you go from the defending state of the game to any other state. If you read up on the RandomEncounterText() function, you'll want to use it here.



    HandleSpare() Happens when you select the Spare option from the Mercy menu, regardless of whether a monster is spareable or not. This event fires after the sparing of monsters is completed. If you spare the last enemy in the encounter, this function will not happen - the encounter is over at that point.



    HandleItem( item_ID, position) Happens when you select an item from the item menu.

    • item_ID: The name of the item used, converted to all-caps. Similar to HandleCustomCommand in monster scripts.
    • position: The position of the item used in the player's inventory. The first item is number 1.

    In CYF, you can use the Inventory object to edit the player's inventory. The items' names will be in caps, like with HandleCustomCommand(). 
    function HandleItem(ItemID, position)
    if ItemID == "DOGTEST2" then
        BattleDialog({"You selected The Second Dog.", "You are truly great."})
    else
        BattleDialog({"You didn't select The Second Dog.", "You could've picked better."})
    end
    DEBUG("You chose item #" .. position .. " in your inventory.")
end



    OnGameOver() This function runs the instant the last living player dies.
    But this event works differently than the others.

    By adding return false, you can prevent the game over sequence from starting and replace it with your own code!

    NOTE: If you plan to use BattleDialog in this event, you MUST call State("NONE") first!

    WARNING: Be careful when using return false to run your own code.
    CYK is designed to work with living, active players. You may encounter errors and problems outside of ENEMYDIALOGUE and NONE!

    ALSO NOTE that killing the players again will trigger this event a second time!

    States




    EnteringState( newstate, oldstate) A new, more flexible way of handling state changes. When you enter a new state, this function will fire with newstate containing the new state's name, and oldstate containing the previous state's name. Both are in all caps. One of the most powerful things about it is that you can use State() here to interrupt state changes initiated by the engine itself.

    Calling State from within this function will call EnteringState again!


    Possible states and when they execute are below:

    • ACTIONSELECT- Returning to the main part of the battle, where you can select FIGHT/ACT/ITEM/MERCY.
    • ATTACKING- When the Players are attacking the enemies.
    • DEFENDING- When the enemy/enemies finish dialogue, and one or more waves start.
    • ENEMYSELECT- When you've selected either FIGHT or ACT, and need to select an enemy.
    • ACTMENU- When you've selected an ACT target, and must now select an ACT command.
      This is also the state used when a Player is selecting a SPELL to cast.
    • ITEMMENU- When you've selected ITEM.
    • MERCYMENU- When you've selected MERCY.
    • ENEMYDIALOGUE- When your enemy/enemies start their dialogue.
    • PLAYERTURN- When you execute all of the Players' actions. Entered when the last Player's action is selected.
    Note: There are four states not mentioned here - because they never occur naturally. You can find a more in-depth explanation of States in the following function.

    For a clearer example, here's a code snippet replicating the older events above. 
    function EnteringState(newstate, oldstate)
    if newstate == "ENEMYDIALOGUE" then
        --same as EnemyDialogueStarting()
    elseif newstate ~= "ENEMYDIALOGUE" and oldstate == "ENEMYDIALOGUE" then
        --same as EnemyDialogueEnding(). Alternatively, check for newstate == "DEFENDING"
    elseif newstate ~= "DEFENDING" and oldstate == "DEFENDING" then
        --same as DefenseEnding()
    end
end


    State( state_to_go_to, arguments...) [E/M/P/W] A powerful function that immediately skips the battle to the specified state, rather than following the default conventions. Below is a list of valid strings you can pass to it, what state you'll be going to and what that state entails. The states you pass to it are case-invariant, but uppercase is recommended for readability.


    State may also need several other arguments to determine some data related to the states it tries to go to. This data allows the engine to specify which Player is choosing an action, or to specify which part of a state we are in.

    Here is a list of states CYK can enter and what complementary arguments it needs in order to switch to the selected state correctly:


    • SETUP - The zeroeth state of an encounter. SETUP takes care of the transition between the Overworld and Encounter interfaces: It builds the encounter's UI, Entities and Background, then triggering the INTRO state. (or skipping to the ACTIONSELECT state if you set skipintro to true.)

    • INTRO - This is the first state CYK is in. It hides the main UI of the battle and the TP bar, and plays the "Intro" animation of all the Players. The state ACTIONSELECT is entered once all of the Players' animations are done.

    • ACTIONSELECT - While in this state, you can select FIGHT/ACT/ITEM/SPARE/DEFEND for the current Player.

      Needs the id of the Player as argument #1. Must be a number between 1 and the max number of players.

      * If you're not sure, this is the position of the player in players, or the value of Entity.ID.

    • ENEMYSELECT - This state displays a list of all active enemies in the encounter. This state gets entered automatically after choosing FIGHT, ACT or SPARE.

      Needs the id of the Player as a argument #1 (same as ACTIONSELECT).
      Also needs a a string as argument #2:
      • "FIGHT" - Acts the same as manually choosing the FIGHT button.

      • "ACT" - Acts the same as manually choosing the ACT button.
        Will throw an error if this Player doesn't have an ability named "Act".

      • "SPELL" - Acts the same as choosing a target for a spell. Will throw an error if this Player has an ability named "Act".

        Needs a third parameter which is the name of the spell this Player will use as a string.

      • "ITEM" - Acts the same as choosing the target for an item.
        Will throw an error if this item is supposed to target all the Players or all the enemies.

      • "SPARE" - Acts the same as manually choosing the SPARE button.

    • ACTMENU - This state displays all of the options from the selected enemy's commands in order if the current Player has an ability named "Act", otherwise it will display the different spells of the Player.

      Needs the id of the Player as argument #1 (same as ACTIONSELECT).
      Also needs the id of the Enemy as argument #2. Must be a number between 1 and the max number of enemies.

    • ITEMMENU - This state shows all of the player's available items. It's tied with the HandleItem function (see Game events for more information).

      Needs the id of the Player as argument #1 (same as ACTIONSELECT).

    • PLAYERTURN - This is the state where all of the Players' actions are executed, one after the other. Entered when the last Player's action is selected.

    • ATTACKING - Starting this state will put you in the screen where you either deal damage to an enemy by pressing "Z" at the right time, or you wait and miss entirely, for each Player who selected the FIGHT button.

      The enemies each Player attacks is the same as the enemy selected during their own turn in the state ENEMYSELECT. After this state ends, the encounter moves on to the ENEMYDIALOGUE state by default.

    • ENEMYDIALOGUE - When starting this state, currentdialogue gets read from the Encounter script for every monster and their dialogue is displayed. If that doesn't exist, it'll return something at random from the randomdialogue list.

    • DEFENDING - When starting this state, nextwaves is read out from the Encounter script and all scripts in the Waves folder with matching names will be part of this defense step. wavetimer is also read from the Encounter script at this time.

    • BEFOREDONE - This is the state that is entered whenever there are no more active enemies. It displays the Players' "EndBattle" animation and the amount of D$ and EXP they won. This state can't be exited as long as there are no active enemies.

    Finally, there are two "special" states that evoke behavior in the engine itself:


    • DONE - Changing state to DONE will instantly force the player to the mod selection screen.

    • NONE - This state does nothing. It is entered for the first frame of the encounter, but entering it manually will completely freeze your encounter. It might be useful if you want to disable all of Unitale/CYF's basic functionality.



    Freezing states As of CYF v0.6.2.1, calling State("PAUSE") will perfectly "pause" an encounter. The last active state will remain active, but in a frozen state, until you call State again.

    * Only the Update function of the Encounter script will remain active here.

    * To unfreeze a state, you must use State to switch to the last active state.     GetCurrentState() will tell you the name of the frozen state.


    GetCurrentState() returns [E/M/P/W] Returns the name of the current state (see above for all states).
